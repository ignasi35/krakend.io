KrakenD - Rate limiting Documentation

The rate limits allow you to restrict the traffic to any component of the stack (the KrakenD itself and the actual API backend services) and mainly cover two different purposes:

  • Avoid flooding your system with massive requests
  • Establish a quota of usage for your API
  • Create a simple QoS strategy for you API

This measures are complementary to the Circuit Breaker.

Rate limits levels

There are three different levels where you can apply throttling limits, if the same limit is defined in several levels the most specific takes place. The levels are:

  • Service level (Global)
  • Endpoint level
    • Client level
  • Backend

Service and Endpoint levels are always defined at service configuration level (file /etc/krakend/service.yml), while the Client level is defined at Application configuration (krakend.json or UI).

Global level

The global_max_rate is the hard limit of the KrakenD service (defaults to 0). This is similar to what you would do at a firewall level to limit/burst the traffic you send to your backend. Leaving this value set to 0 makes KrakenD go as fast as possible. If you want to set a limit a good reference is to chose the 95% of your backend throughput.

When set, the rate counts against all endpoints.

You can find this parameter in the service.yml configuration file, e.g:

global_max_rate: 1500 # req/s

Default max rate per endpoint

HARD LIMIT At a service level you can define a default maximum rate limit per endpoint. Unless the application configuration overwrites this limit this limit will be set for all the endpoints.

Limits the request per second the KrakenD service will process for a single endpoint.

This parameter is defined at the service.yml configuration file:

max_rate_per_endpoint: 1000 # req/s

Client

Limits the request per second the KrakenD service will process for a single endpoint and a given client. There are two client identification strategies: by token set in a request header and by client IP.

This level is defined at the krakend.json configuration file:

...
{
  "endpoint": "/account",
  "backend": [
    {
      "url_pattern": "/user"
    }
  ],
  "client_max_rate": "1"
},
{
  "endpoint": "/products",
  "backend": [
    {
      "url_pattern": "/items/list"
    }
  ],
  "client_max_rate": "1",
  "throttling_header": "X-Token"
},
...

Client rate limiting could have an impact on the global performance, so use it with caution.

Backend

Limits the request per second the KrakenD service will send to a given backend for a given endpoint.

This parameter is defined at the krakend.json configuration file:

...
{
  "endpoint": "/github",
  "backend": [
    {
      "url_pattern": "/"
    }
  ],
  "endpoint_max_rate": "1"
},
...

Implementation details

Our rate limiting middlewares are using a token bucket algorithm with a bucket capacity == tokens added per second so KrakenD is able to allow some bursting on the request rates.

Limiting the amount of connections per client (client_max_rate)

On the other hand, the client_max_rate limits the maximum requests allowed to every client within a second. The value should be significantly lower than you would use for max_rate, as this limit will be multiplied by the number of active clients in that moment.

Example:

client_max_rate : 10

If 200 different customers (with different IPs) hit the KrakenD, they will be allowed to generate a total traffic of

200 ips x 10 req/s = 2000req/s

Comparison of max_rate vs client_max_rate

The max_rate is an absolute number and you have the exact control on how much traffic you are allowing to hit the backend. In an eventual DDoS the max_rate can help in a way since it won’t accept more traffic than allowed. But on the other hand a single host could abuse the system taking a big percentage of that quota.

The client_max_rate is a limit per client. It won’t help you if you just want to control the total traffic, as the total traffic supported by the backend depends on the number of different requesting clients. A DDoS will then happily pass through, but on the other hand you can keep any particular abuser limited to its quota.

Depending on your use case you will need to decide if you use one, the other, the two, or none of them (fastest):

Performance note!

From a performance point of view the client_max_rate drops the performance as every incoming client needs to be monitored.
